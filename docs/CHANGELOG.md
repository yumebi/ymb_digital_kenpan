# 作業履歴

`DigitalKenpan.jsx` の開発経緯です。バージョン番号はローカルのバックアップファイル名(`DigitalKenpan_vN.jsx.bak`)に対応しています。

## v1 — 初版

- チェック項目25件を実装(サイズ/トンボ、フォント、カラー、画像、オブジェクト・効果、その他)
- 事前設定(仕上がりサイズ・印刷カラー数・各種閾値)、設定の保存/読込
- 結果画面(ツリー表示・検出オブジェクト一覧・選択してズーム)、HTML/CSVレポート出力
- 参考: Illustrator用プリフライトツール([IllustratorPreflight](https://github.com/sumitkumar511/IllustratorPreflight))、Acrobatプリフライトプロファイル相当の項目を実現可能な範囲で移植

## v1 → v2 — フリーズ修正・中断機能

- **フリーズ修正**: リンク画像のヘッダ解析が画像1点につき最大1MB・全バイトループで読んでいたのが原因。先頭64KBのみ読む方式に変更し、カラーモード/解像度チェック間で解析結果をキャッシュ共有
- 検版実行中の **ESCキー長押しによる中断**、設定画面への **キャンセルボタン** を追加
- ロック済み/非表示レイヤーにあるオブジェクトを選択してズームできない問題を修正(自動でロック解除してから選択)

## v2 → パレット化の試行 → v2へロールバック

- 結果画面をモーダルダイアログから非モーダルパレットに変更し、開いたままIllustrator本体を操作できるようにする改修を実施
- Windows/macOS 両方で動作不良(ドキュメント同一性判定 `===` の不具合、ウィンドウ連携の破綻)が発生したため、**作業ログから差分を再生してv2の状態へロールバック**
- 教訓: ウィンドウ構造の大改修と機能追加を同時に行わず、小さい差分で段階的に検証すること

## v2 → v3 — 設定画面のスクロール対応

- 設定画面をウィンドウサイズより縮めると項目やボタンが隠れる問題に対応し、縦スクロールバーを実装
- リサイズドラッグ中に全コントロールを毎回再レイアウトしていたことによる「引っかかり」を解消(確定時のみ再計算)
- スクロールバーが出現しない不具合・リサイズを繰り返すとボタン部の余白が増え続ける不具合を修正

## v3 → v4 — macOS向け表示・操作性の修正

- macOSで進捗表示のファイル名が表示されない問題を修正(ラベル幅の確保・ファイル名の短縮表示)
- 検出オブジェクトへの「選択してズーム」で倍率が異常値(500%超・10%未満など)になる問題を修正し、50〜400%にクランプ

## v4 — GitHub公開

- リポジトリを [yumebi/ymb_digital_kenpan](https://github.com/yumebi/ymb_digital_kenpan) として公開
- README・仕様詳細(`docs/SPEC.md`)・本履歴を整備

## v4 → v5 — macOS表示不良の修正(設定画面ボタン・スクロールバー・進捗ファイル名)

- **設定画面のボタン列・スクロールバーが表示されない件**: ウィンドウの初期自動サイズがmacOSでは画面内に収まりきらず、下部のボタン列・スクロールバーが可視領域外に押し出されていたとみられる。初期レイアウト確定のタイミング待ち、明示的なウィンドウサイズのクランプに加え、原因の特定に依存しない最終防衛策(ボタン列が画面内に収まっているか直接検証し、はみ出ていれば強制的に引き戻す)を追加
- **進捗表示のファイル名が表示されない件(v4の対策では未解決)**: 幅確保に使っていた `characters` プロパティが実は表示用コントロールでは正式サポート外だったことが判明。表示コントロールを変更し、公式にサポートされる方法で幅を確保する形に修正

## v5 → v6 — バージョン表示・設定画面ボタンの配置変更

- アプリ内(タイトルバー・HTML/CSVレポート)に**バージョン番号を表示**(現在 v1.6.0)。以後の修正のたびに値を更新する運用とする
- v5のスクロールバー・ボタン非表示修正がMac実機で依然解消しなかったため方針転換。**ボタン列(設定を読込/設定を保存/キャンセル/検版実行)を設定画面の一番上**(ドキュメント名表示のすぐ下)に移動。ウィンドウの最上部はレイアウト計算がどうズレても必ず画面内に入るため、位置計算に依存せずボタンの可視性を構造的に保証する

## v6 → v7 — 結果画面のレイアウト変更

- 結果画面のボタン列(設定に戻る/レポート保存(HTML)/レポート保存(CSV)/閉じる)を、設定画面と同じ理由で**画面最上部**に移動
- 検出オブジェクト一覧の表示枠を拡大(項目一覧との境界をドラッグ調整できる仕組みはScriptUIでの安定実装に確信が持てず見送り、固定で大きめのサイズを確保する形に)
- 右ペインの表示順を「検出オブジェクト一覧 → 選択してズームボタン → 説明欄」に変更(以前は一覧の下に説明欄、その下にボタンだった)

## v7 → v8 — 結果画面の余白詰め・パネル分割・視認性向上

- ウィンドウを縦に伸ばすと検出オブジェクト一覧・ボタン・説明欄の間隔が広がっていく問題を修正。伸びる分は一覧だけが吸収し、ボタン・説明欄は常に一覧のすぐ下に密着するように
- 項目一覧(ツリー)と検出オブジェクト一覧の境界を**ドラッグで調整できるスプリッター**を実装。ドラッグ操作の信頼性が不安な場合の保険として、段階的に幅を調整する「◀ 項目一覧を広げる / 検出オブジェクト一覧を広げる ▶」ボタンも併設
- 「原因と対応」の説明欄をアンバー色の太字に変更し視認性を向上

## v8 → v9 — 段階調整ボタンの削除

## v9 → 診断ログ機能の追加(バージョン番号は据え置き)

- macOSでのスクロールバー・進捗ファイル名の非表示が推測ベースの修正では解決しきれなかったため、実行時の状態をテキストファイルに書き出す診断ログ機能を追加。`DigitalKenpan_debug.log`(デスクトップ、書き込み不可時は書類フォルダにフォールバック)にスクロールバー計算・進捗ラベル更新の実測値を記録する
- `KENPAN_DEBUG_LOG = true` をファイル冒頭で切替可能。問題解決後は false にして無効化する想定
- 次にmacOSで実行した際のログを確認してから、当て推量ではなく実測値に基づいて修正する方針に切り替えた

## 診断ログの結果に基づく確定修正(v1.10.0)

Macから取得した診断ログを解析し、2件とも実測値に基づく確定原因が判明した(推測ベースの修正ではない)。

- **スクロールバー非表示**: 内部計算・可視性の設定自体は正しく動作していた(ログで確認)。原因はプロパティ変更後に親コンテナへの明示的な再レイアウト呼び出しが無かったこと(macOSではプロパティ変更だけでは再描画されないことがある)。可視状態が変化した時だけ再レイアウトを呼ぶよう修正
- **進捗ファイル名の空欄**: 表示側の問題ではなく、`file.displayName` がMac環境で空文字列を返していたことが原因(ログで実証)。`displayName → name → fsName` の順で試すフォールバックチェーンに変更
- 原因究明に使った `KENPAN_DEBUG_LOG` は既定値を false に戻した(コード自体は温存、再調査時にtrueへ)

## 診断ログに基づくMac確定修正(v1.10.0〜v1.13.0)

Macから取得した診断ログの実測値を解析し、複数の不具合の確定原因が判明・修正された。

- **スクロールバー非表示**: 画面が小さいMacでウィンドウが自然幅より縮小されても内部コンテンツが自然幅のままで、スクロールバーがウィンドウ右端の外側(画面外)に配置されていたことが原因。横方向もウィンドウ実幅から絶対計算する方式に変更(v1.12.0)
- **進捗ファイル名の空欄・文字化け**: `file.displayName` がMacで空文字列を返す問題は `name`/`fsName` へのフォールバックで解決。フォールバック先の `file.name` がURIエンコード名を返す仕様(日本語が`%E3%81%82...`表記になる)だったため `decodeURI` を適用(v1.10.0, v1.12.0)
- 結果画面にも同様のウィンドウフィット処理を適用し、下部要素(選択してズーム等)が画面外に切れる問題に対応(v1.13.0)

## UI改修(v1.14.0〜v1.15.0)

- スクロールバーをウィンドウ右端に密着させ、枠線を減らして背景色で区切る「面」のデザインに変更(v1.14.0)
- **スクロールバーを常時表示に変更**(スクロール不要時はグレーアウトのみ)。可視性トグルによる再レイアウト誘発が無くなり、ウィンドウ右側・ボタンのガタつきを解消(v1.15.0)
- 設定画面に横スクロールバーを追加(v1.15.0)
- 設定画面・結果画面のボタン列を右寄せから左寄せに変更(v1.15.0)
- 結果画面を「ウィンドウに追従して伸縮」から「固定サイズのコンテンツ+スクロールで表示範囲移動」方式に作り替え(設定画面と同じ仕組み)。項目一覧とオブジェクト一覧の幅比率調整(スプリッター)は維持(v1.15.0)

## v1.15.0のロールバックと再修正(v1.16.0)

v1.15.0(スクロールバー常時表示化・横スクロール追加・ボタン左寄せ・結果画面の固定サイズ化)を実機で試したところ挙動がおかしくなったため、一旦v1.14.0の状態に巻き戻して別ブランチ(`diag/v15-windows`)で原因調査を実施。

**確定原因**: `win.layout.layout(true)` の呼び出しが `win.onResize` を再度同期的に発火させ、フィット処理→再描画→再発火…という再入ループになっていた。ウィンドウが操作していないのに際限なく広がり続け、最終的には同一タイムスタンプ内で処理が無限に繰り返される完全なフリーズ状態に陥っていた(Windows実機の診断ログで実証)。

**修正**: 再入防止フラグ(`FIT_IN_PROGRESS`)を導入し、`win.layout.layout(true)` を呼ぶ危険な処理区間中は `win.onResize` の再入を確実にブロックする設計に変更。修正後の実機ログでフリーズが解消されたことを確認し、v1.16.0としてmainに統合した。

## リサイズを繰り返すとウィンドウが伸び続ける問題の修正(v1.17.0)

v1.16.0でJSレベルの無限ループ(フリーズ)は解消したが、「リサイズするたびに伸びる」という別の症状が残っていた。

**確定原因**: `win.layout.layout(true)`(ウィンドウ全体を対象にしたlayout呼び出し)を呼ぶと、Windows環境ではその副作用として `win.size` 自体が+25px前後増加することが実機ログで判明した。スクロールバーのenabled切替のたびに呼ばれる高頻度パス(`forceScrollbarReflow`/`forceResultScrollbarReflow`)がこの呼び出しを含んでいたため、リサイズを繰り返すたびに増加が積み重なっていた。

**修正**: ウィンドウ全体を対象にした `layout.layout(true)` 呼び出しを、高頻度パス(スクロールバー再描画・スプリッタードラッグ・自然サイズ測定)から完全に排除し、個別コンテナ単位の呼び出しに置き換えた。画面切替・起動時など低頻度の箇所にのみ限定して残した。

## UI磨き込み(v1.18.0〜v1.19.0)

- 横スクロールバーをウィンドウ下端に密着させた(縦スクロールバーの右端密着と同じ考え方)。設定画面・結果画面の両方に適用
- 結果画面のステータス欄・説明欄で長文が文中で途切れる問題を修正。項目ごとに文章量が大きく異なる(1文で終わるものもあれば複数文にわたるものもある)ため、固定高さの積み増しではなく**表示するテキストの文字数から必要な高さを都度計算する方式**に変更(v1.19.0)
- v1.19.0の見積もりでも最後の行がわずかに欠ける不具合が実機で確認されたため、行高・文字幅・パディングの各係数を余裕を持たせた値に変更し、さらに全体へ+20%のバッファを追加。次に欠けた場合に一発で校正できるよう、見積もり高さと実際の描画サイズを両方ログに記録するようにした(v1.20.0)
- v1.20.0のログで「見積もり高さは正しく計算されているのに実際の描画サイズが常に固定値のまま変わっていない」ことが判明。既に一度レイアウト済みのコントロールは`preferredSize`の再設定だけでは`.size`が更新されないというScriptUIの癖が原因で、`.size`を直接上書きするよう修正(v1.21.0)
- 検版実行を開始するとリサイズ済みのウィンドウがデフォルトサイズに戻ってしまう不具合を修正。原因はv1.17.0で危険と判明していた「ウィンドウ全体を対象にしたlayout呼び出し」の見落としが1箇所残っていたため(v1.21.0)
- 項目一覧と検出オブジェクト一覧の境界(スプリッター)が完全に無反応だった問題に対応。ScriptUIはグループ要素へのマウスイベントがほぼ未サポートのため、より確実にサポートされるウィンドウ自体にイベントを付け替え。検出オブジェクト一覧の初期幅も拡大(v1.21.0)
- スプリッターのドラッグはウィンドウ単位のイベントに変更しても実機で反応しなかったため、ドラッグ機能自体を削除(静的な区切り線として残す)。幅は可変調整を諦める代わりに固定値を全体的に見直し、項目一覧260px・検出オブジェクト一覧540pxに拡大(v1.22.0)
- v1.21.0の`.size`直接上書き修正でも、選択項目によっては下部要素(ステータス欄)が欠ける不具合が残っていた。原因は、`.size`直接代入は個別コントロール自身の描画には反映されるが、親(resultBody)の「必要サイズ計算」にはScriptUIレイアウトエンジンが反映しないという別の癖だったため、直近に適用した高さと初期値の差分を自前で加算して親の必要高さを補正するよう対応(v1.23.0)

## 既知の課題(次の対応予定)

- v1.22.0でスプリッターのドラッグ機能自体を削除したため、項目一覧/検出オブジェクト一覧の幅比率はユーザー調整不可(固定260px/540px)。将来的にドラッグでの調整が必要になった場合は別の実装方式を検討する必要がある
- v1.23.0の差分補正は「ScriptUIレイアウトエンジンの癖に対する自前ワークアラウンド」であり、根本修正ではない。resultBody配下に新たな可変高さ要素を追加する場合は同様の補正が必要になる
- v5〜v8の macOS 向け修正は実機未検証(手元にmacOS実行環境が無いため、ロジック上の対策に留まる)
- パレット化(検版ツールを開いたままIllustrator本体を操作できるようにする)は再挑戦の余地あり。ドキュメント同一性判定をファイルパス比較に変える方針で、単独の小さな差分として再実装する想定
